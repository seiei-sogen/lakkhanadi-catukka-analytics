# 既知の制限（v0.1.0 MVP 時点）

本 Skill の v0.1.0（MVP）で把握されている制限事項。今後のバージョンで対応する可能性があるが、MVP のスコープには含まれない。

## 機能スコープ外（要件定義 §Out of Scope より）

- **ファイル取り込み機能未対応**: 入力はテキスト引数のみ。PDF・Word・大量ファイルの一括分析は未対応。
- **多言語出力非対応**: 出力は日本語固定（REQ-404）。英語・中国語などの出力は未対応。
- **部分出力モード非対応**: 「lakkhaṇa だけを出力」のような部分出力は未対応。常に 4 セクション網羅で出力する（REQ-001）。
- **出典・根拠の能動的表示なし**: `lakkhanadi-catukka-example.ncl` を内部参照する明示的なクエリ機構はない。LLM の解釈に委ねる。

## プラットフォーム関連

- **前処理フック非対応環境での挙動**: Claude Code skills と Codex CLI skills のどちらも 2026-05 時点で前処理フック機構を公式に持たない。そのため `scripts/normalize.rs`（Rust 補助スクリプト）は実運用での自動起動経路を持たず、Pali 表記正規化は SKILL.md 内テーブル（「2. Pali 表記正規化」）が主経路となる。`normalize.rs` は (a) ルール表のテストハーネス、(b) 将来ホストが前処理フックを提供したときの実装、として残す。
- **Codex CLI 仕様の流動性**: 本 ADR（`docs/governance/adr_skill_platform.md`）は 2026-05 時点の `developers.openai.com/codex/skills` 公式ドキュメントを根拠としている。仕様変更があった場合、リリース直前または Codex 公式 changelog で skills 関連変更が告知されたときに ADR と SKILL.md / `agents/openai.yaml` / README を整合させる必要がある（更新責任者: 本リポジトリのメンテナ）。
- **配置パスの自動化**: ホスト側が `skills/<name>` の集約配置を直接読めるようになれば、シンボリックリンク運用は不要になる見込み。MVP は手動コピーまたはシンボリックリンクで対応。

## 検証関連

- **実機検証はユーザー手動実行**: TASK-0019（Claude Code）／TASK-0020（Codex CLI）の skill 認識・明示呼び出し・7 サンプル動作の最終確認は、開発環境では再現困難なため、**実機での手動実行**で完了する。手順は `docs/tasks/.../verification/{claude-code,codex-cli}/README.md` を参照。
- **judge 評価の実行コスト**: 7 サンプル × 8 観点 = 56 判定を含む LLM-as-judge は API コストと時間を要するため、CI での自動実行は MVP スコープ外。リリース前または重要変更時の**手動バッチ実行**を推奨（手順: `skills/lakkhanadi-catukka/tests/judge/run.md`）。
- **judge 自体のばらつき**: LLM 推論の確率的性質上、同一入力に対する verdict が完全に一致しない可能性がある。`run.md` ステップ 8 で 3 回実行による一貫性確認を行うが、軽微な揺れは許容する設計。揺らぎが大きい観点が見つかれば SKILL_JUDGE.md の判定指示を強化する（TASK-0017 へフィードバック）。

## 安全性関連

- **観察的捉え直しの機械的判定限界**: 「鬱」→「いま感じている重さ」のような観察的捉え直し（REQ-106）は LLM の解釈に依存する。代表的な対応表（SKILL.md「4.3 対象の観察的捉え直し」）を提供しているが、対応表に無い対象では LLM が自然言語で対応する。極端な例では捉え直しが浅くなる可能性がある。
- **専門家相談促進の文言**: medical / legal / religious カテゴリでは末尾に固定文言を添える設計（REQ-201）。文言のカスタマイズ（職種・国・状況に応じた具体化）は MVP 範囲外。

## 出力品質の境界

- **長さモード境界条件**: 短縮モードで詳説を完全省略した場合、要約表 + 一言まとめのみが出力される。「詳説の最小 3 文」（EDGE-101）は通常モード以上で適用され、短縮モードに限り省略許容（dataflow.md フロー 3 備考）。
- **古典 vs 応用判定の保守性**: 判定に迷う場合は **applied（応用扱い）にフォールバック**する設計（REQ-406「教義として確認できない内容を断定しない」と整合）。これにより、古典明示トピックに応用マーカーが誤付与される可能性は理論上ある。代表例リスト（SKILL.md「5.1」）でカバーされる主要トピックでは発生しない見込み。
- **paccupaṭṭhāna の三面網羅**: 身体感覚／思考・感情／世界の見え方の三面（REQ-008）の網羅は最重要観点として SKILL.md「6.1」と SKILL_JUDGE.md 観点 2/3 で担保するが、短縮モードでは三面いずれかへの簡潔な言及で許容する。

## 補助スクリプト関連

- **`normalize.rs` の文字境界**: case-insensitive マッチに `to_lowercase()` を用いており、ASCII / 共通の Pali 文字では問題ないが、稀に Unicode の特殊な大文字小文字対応（Turkish dotted-I 等）でバイト長が変わるケースは未対応。MVP のスコープでは発生しない見込み。
- **「推測まで」の正規化は不採用**: ルール表に該当しない極端な崩れ（"padaaaaana" 等）は正規化せず通す。これは PRD ヒアリングRound 3 の方針で確定。`unknown_term` として LLM に委ねる設計。

## 配布・運用関連

- **配布形態は本リポジトリのみ**: crates.io 公開・GitHub Release 配布物の作成は MVP 範囲外。ユーザーはリポジトリをクローンして配置する運用。
- **CHANGELOG・タグ運用**: `v0.1.0` を MVP リリースタグとして用意するが、自動デプロイ・自動 Release Notes 生成は未整備。
