# lakkhanadi-catukka-analytics-requirements ヒアリング記録

**作成日**: 2026-05-05
**ヒアリング実施**: step4 既存情報ベースの差分ヒアリング

> **2026-05-07 注記**: 本記録中の「TypeScript / TS」は、後の REQ-403 改訂で **Rust** に変更された。原文は当時の議論をそのまま残す。最新の実装言語は `codd/codd.yaml`・`docs/spec/.../requirements.md` REQ-403 を参照。

## ヒアリング目的

PRD（既存 GPTs プロンプト = `docs/requirements/requirements.md` / `spec.md`）と codd ワークフロー設定（`codd/codd.yaml`）から把握できる事項のうち、**Skill 化に当たって未定義・曖昧だった以下のカテゴリ**について、4 ラウンド・計 15 問のヒアリングを実施した。

- ホスト環境 / 配布形態
- 実装の主体（プロンプト vs コード）
- 起動条件 / 出力モード切替方法
- 入出力仕様
- 拒否ポリシー / 正規化スコープ
- テスト戦略
- リポジトリ構造 / MVP スコープ

## 質問と回答

### Round 1: 基本仕様

#### Q1-1: Skill のホスト・配布先

**質問**: Claude Code skills / Codex CLI skills / 両方対応 / ツール独立の MD ファイル
**カテゴリ**: 既存設計確認（codd ADR と README の「Codex skills」言及の整合）
**背景**: README では「Codex の skills」、codd Wave 1 では「Codex skill platform」とあるが、2026 年現在 Claude Code にも skill 機構があるため両対応の可能性を確認した
**回答**: **両方対応**
**信頼性への影響**: REQ-402（両プラットフォーム対応）が 🔴 → 🔵、NFR-301（配置構造）の方針が固まる

---

#### Q1-2: 実装の主体

**質問**: プロンプト・markdown 中心 / TypeScript と並立 / TypeScript 中心
**カテゴリ**: 既存設計確認（codd `language: typescript` との整合）
**背景**: codd では language: typescript と宣言されているが、skill は一般にプロンプト markdown であり、ここに矛盾の可能性があった
**回答**: **プロンプト・markdown 中心**
**信頼性への影響**: REQ-403（実装形態の制約）が 🔵 で確定。codd の `language: typescript` は「補助スクリプトを書く場合の言語」と解釈してよい

---

#### Q1-3: 出力言語方針

**質問**: 日本語固定 / 入力言語に追従 / ユーザー指定可
**カテゴリ**: 未定義部分の詳細化
**背景**: 既存 PRD は日本語で書かれているが、明示的な「日本語固定」要件はなかった
**回答**: **日本語固定**
**信頼性への影響**: REQ-404（出力言語固定）が 🔴 → 🔵、MVP スコープ外項目「複数言語出力」が確定

---

#### Q1-4: 構造化リファレンス（lakkhanadi-catukka-example.ncl）の使い方

**質問**: 常に参照 / オンデマンド参照 / 参考資料として同梱のみ / 使わない
**カテゴリ**: 未定義部分の詳細化
**背景**: 構造化リファレンス（旧名 `four-sample.md`、現行 `lakkhanadi-catukka-example.ncl`）がリポジトリにあるが、skill 実行時の利用方針が未定だった
**回答**: **オンデマンド参照**
**信頼性への影響**: 古典トピック判定時のみ参照する分岐が確定。MVP スコープ外「出典・根拠表示機能」と整合

---

### Round 2: 振る舞いの詳細

#### Q2-1: 起動トリガー

**質問**: 明示呼び出しのみ / キーワードトリガー / 広いトリガー
**カテゴリ**: 未定義部分の詳細化
**背景**: Claude Code skills では description フィールドが LLM の自動起動判定に使われるため、トリガー方針を確認
**回答**: **明示呼び出しのみ**
**信頼性への影響**: REQ-409（明示呼び出し）が 🔴 → 🔵

---

#### Q2-2: 出力長さモードの切り替え

**質問**: テキストシグナルで自動判定 / 明示パラメータ / 両方
**カテゴリ**: 既存設計確認（PRD「出力の長さ」節の解釈）
**背景**: PRD では「短く / 詳しく」のシグナルで切り替えるとあるが、skill 起動時のパラメータ受け取り方も検討する必要があった
**回答**: **テキストシグナルで自動判定**
**信頼性への影響**: REQ-101〜103（長さモード分岐）が 🔵 で確定

---

#### Q2-3: 複数対象の分析・比較

**質問**: 不要（1依頼1対象） / 並列分析 / 比較モードあり
**カテゴリ**: 追加要件
**背景**: GPTs の既存プロンプトは 1 対象前提だが、skill 化に伴って複数対象を扱いたい需要があるか確認
**回答**: **不要（1 依頼 1 対象）**
**信頼性への影響**: EDGE-004（複数対象入力時の挙動）が 🔵 で確定。MVP スコープが小さく保たれる

---

#### Q2-4: 応用的分析マーカーの付与ポリシー

**質問**: 閾値ベース自動 / 常に明記 / 明示要求時のみ
**カテゴリ**: 既存設計確認（PRD「重要な方針」の運用化）
**背景**: PRD では「仏教教義として確認できない内容は『応用的分析』と明記する」とあるが、自動判定の粒度が曖昧だった
**回答**: **閾値ベースで自動**
**信頼性への影響**: REQ-104, REQ-105（応用マーカーの分岐条件）が 🔵 で確定

### Round 3: 入出力 / 拒否 / テスト

#### Q3-1: 入力の受け付け方針

**質問**: テキストのみ・無制限 / テキストのみ・推奨長さあり / テキスト+ファイルパス
**カテゴリ**: 未定義部分の詳細化
**背景**: skill が外部リソース（ファイル）を読みに行くか否かは、tools 利用範囲に直結するため確認
**回答**: **テキストのみ・長さ制限しない**
**信頼性への影響**: NFR-001（無制限入力）と REQ-408（tools なし）が 🔵 で確定。MVP スコープ外「ファイル取り込み」と整合

---

#### Q3-2: 拒否・軽減ポリシー

**質問**: 拒否せず警告と捉え提示 / トピックごとに拒否 / 拒否しない
**カテゴリ**: 既存設計確認（PRD「重要な方針」「注意」の運用化）
**背景**: PRD は「医療・法律・宗教の権威として断定しない」とあるが、要求自体を拒否するか / 受けて警告で凌ぐかが曖昧
**回答**: **拒否せず警告と捉え提示で分析**
**信頼性への影響**: REQ-106（警告 + 観察的捉え直し）が 🔵 で確定。これは PRD のトーン（観察的・非難しない）とも合致

---

#### Q3-3: パーリ語正規化のスコープ

**質問**: 既知リストのみ / 既知リスト+IAST揺れ / 推測まで
**カテゴリ**: 既存設計確認（codd Wave 3「normalizer-rules」の細粒度）
**背景**: PRD は padahāna / paccupahāna の 2 ケースのみ明示しており、IAST やダイアクリティカル揺れまで対応するかが曖昧
**回答**: **既知リスト + IAST とダイアクリティカル表記揺れ**
**信頼性への影響**: REQ-108（IAST 正規化）が 🔴 → 🔵 として明示要件化

---

#### Q3-4: テスト・品質保証の方針

**質問**: 文例集+手レビュー / LLM-as-judge / 両方併用 / スコープ外
**カテゴリ**: 追加要件
**背景**: codd Wave 5「test-strategy」ノードはあるが、評価手法は未定義だった
**回答**: **LLM-as-judge で自動評価**
**信頼性への影響**: NFR-302（LLM-as-judge 同梱）が 🔴 → 🔵

**回答**: **MVP に含める**

#### Q4-3: LLM-as-judge テストの MVP 含有

**カテゴリ**: 優先順位

**回答**: **MVP に含める**

#### Q4-4: MVP スコープ外項目の確定

## ヒアリング結果サマリー

### 確認できた事項

- 配布対象は Claude Code skills + Codex CLI skills の両方（REQ-402）
- 実装はプロンプト・markdown 中心、TypeScript は補助に限定（REQ-403）
- 出力は日本語固定（REQ-404）
- 構造化リファレンス `lakkhanadi-catukka-example.ncl` はオンデマンド参照、出典表示は MVP 外
- 起動は明示呼び出しのみ、長さモードはテキストシグナルで自動判定
- 1 依頼 1 対象、複数対象の比較は MVP 外
- 拒否はせず警告と観察的捉え直しで継続（REQ-106）
- パーリ語正規化は既知リスト + IAST 揺れまで対応（REQ-107, 108）
- テストは LLM-as-judge を MVP に含める（NFR-302）
- リポジトリ構造は `skills/lakkhanadi-catukka/` に集約（NFR-301）

### 追加要件（既存 PRD・設計文書になかった項目）

- REQ-108（IAST／ダイアクリティカル揺れの正規化）
- REQ-402（両プラットフォーム対応）
- REQ-404（日本語固定）
- REQ-409（明示呼び出しのみ）
- NFR-301（配置構造）
- NFR-302（LLM-as-judge）
- EDGE-004（複数対象入力時の挙動）

### 残課題

- Skill 用の YAML フロントマター仕様の差分（Claude Code 形式と Codex CLI 形式）— 設計フェーズで吸収（codd Wave 2 system-design）
- LLM-as-judge の判定プロンプトに含める品質チェックリストの粒度 — 設計フェーズ（codd Wave 5 test-strategy）
- 「閾値ベースの応用的分析判定」の具体的な閾値定義 — 詳細設計フェーズ（codd Wave 4 analysis-quality-model）
