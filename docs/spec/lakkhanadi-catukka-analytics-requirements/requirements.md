# lakkhanadi-catukka-analytics-requirements 要件定義書

## 概要

ChatGPT GPTs として運用していた「上座部アビダンマの四相分析（lakkhaṇa, rasa, paccupaṭṭhāna, padaṭṭhāna）アシスタント」を、Coding Agent 向けの skill（Claude Code skills および Codex CLI skills 両対応）として再実装する。任意の対象（概念・経験・感情・現象・文章）を、四相分析の枠組みで整理して日本語で出力する。Skill 本体はプロンプト・markdown を中心に構成する。

## 機能要件（EARS記法）

**【信頼性レベル凡例】**:
- 🔵 **青信号**: PRD・既存設計文書・ユーザヒアリングを参考にした確実な要件
- 🟡 **黄信号**: PRD・既存設計文書・ユーザヒアリングから妥当な推測による要件
- 🔴 **赤信号**: PRD・既存設計文書・ユーザヒアリングにない推測による要件

### 通常要件

- **REQ-001**: システムは、ユーザーが入力した対象（概念・経験・感情・現象・文章）を四相分析の 4 項目（lakkhaṇa, rasa, paccupaṭṭhāna, padaṭṭhāna）に整理して出力しなければならない 🔵 *PRD（spec.md / docs/requirements/requirements.md）「目的」より*
- **REQ-002**: システムは、出力構成を「対象の言い換え（一文）→ 四相分析・要約表（4 行）→ 四相分析・詳説（4 セクション）→ 一言まとめ」の順で生成しなければならない 🔵 *PRD「出力形式」より*
- **REQ-003**: システムは、要約表を「項目 / パーリ語 / 要約」の 3 列構造とし、4 行（特相・作用・現れ方・近因）を必ず含めなければならない 🔵 *PRD「出力形式」より*
- **REQ-004**: システムは、各詳説セクションを最低 3〜5 文で記述しなければならない 🔵 *PRD「納得感を出すための書き方」より*
- **REQ-005**: システムは、詳説セクションに具体例を必ず含めなければならない 🔵 *PRD「納得感を出すための書き方」より*
- **REQ-006**: システムは、出力時にパーリ語仏教用語を使用したら、必ず現代語での言い換えを併記しなければならない 🔵 *PRD「納得感を出すための書き方」・codd Wave 2 規約より*
- **REQ-007**: システムは、接続語（つまり / たとえば / ここで重要なのは）を活用し、wall-of-text にならないように記述しなければならない 🔵 *codd Wave 4 規約「Connective phrases」より*
- **REQ-008**: システムは、paccupaṭṭhāna セクションで「身体感覚」「思考・感情・イメージ」「世界がどう見える感じ」の少なくとも複数の観点を含めて記述しなければならない 🔵 *PRD「paccupaṭṭhāna」節・codd Wave 4 規約より*
- **REQ-009**: システムは、padaṭṭhāna セクションを「根本原因」ではなく「いまそれを支えている近接条件」として、観察的・非難しないトーンで記述しなければならない 🔵 *PRD「padaṭṭhāna」節・codd Wave 4 規約より*
- **REQ-010**: システムは、出力末尾に全体を 1〜2 文でまとめる「一言でまとめると」セクションを必ず含めなければならない 🔵 *PRD「一言でまとめると」より*

### 条件付き要件

- **REQ-101**: 入力テキストに「短く」「簡潔に」というシグナルが含まれる場合、システムは要約表中心の短い出力を生成しなければならない 🔵 *PRD「出力の長さ」・ヒアリングRound 2より*
- **REQ-102**: 入力テキストに「詳しく」「納得感ある感じで」「具体的に」というシグナルが含まれる場合、システムは各詳説セクションを 5〜8 文程度に拡張しなければならない 🔵 *PRD「出力の長さ」・ヒアリングRound 2より*
- **REQ-103**: 入力テキストに長さシグナルが含まれない場合、システムは通常モード（要約表 + 各 3〜5 文の詳説）で出力しなければならない 🔵 *PRD「出力の長さ」より*
- **REQ-104**: 入力対象が仏教古典に明示されたトピック（地水火風の四大、五感の透明色、心所など）である場合、システムは古典に沿った分析を行い、応用的分析マーカーを付けてはならない 🔵 *PRD「重要な方針」・ヒアリングRound 2より*
- **REQ-105**: 入力対象が仏教古典に明示されていない現代的・心理的・技術的トピックである場合、システムは「四相分析の枠組みを応用すると」と明示し、応用的分析マーカーを付けなければならない 🔵 *PRD「重要な方針」・ヒアリングRound 2より*
- **REQ-106**: 入力に医療診断・宗教的断定・他者評価・個人中傷の要求が含まれる場合、システムは「医療・法律・宗教の権威としては断定しない」旨を警告として明示し、対象を「観察可能な現象」として捉え直したうえで分析を継続しなければならない 🔵 *PRD「注意」・ヒアリングRound 3より*
- **REQ-107**: 入力にパーリ語の既知の異形（padahāna, paccupahāna 等）が含まれる場合、システムは標準表記（padaṭṭhāna, paccupaṭṭhāna 等）に正規化したうえで分析しなければならない 🔵 *PRD「重要な方針」・codd Wave 3 規約より*
- **REQ-108**: 入力に IAST／ダイアクリティカル表記の揺れ（lakkhana ↔ lakkhaṇa, thana ↔ ṭhāna 等）が含まれる場合、システムはこれを標準表記に正規化したうえで分析しなければならない 🔵 *ヒアリングRound 3「正規化スコープ」より*

### 状態要件

- **REQ-201**: パーリ語や教義上の厳密性が要求される文脈にある場合、システムは「原典・注釈・信頼できる先生の指導と照合するよう促す」旨を出力に含めなければならない 🔵 *PRD「注意」より*

### オプション要件

- **REQ-301**: システムは、似た現象（例: 「集中」と「執着」）との違いを軽く対比的に記述してもよい 🔵 *PRD「納得感を出すための書き方」より*
- **REQ-302**: システムは、ユーザー入力に含まれる語句・状況・感情を引用して説明に組み込んでもよい 🔵 *PRD「納得感を出すための書き方」より*

### 制約要件

- **REQ-401**: システムは Coding Agent 向けの skill として実装し、ChatGPT GPTs 形式では配布してはならない 🔵 *codd Wave 1 ADR `governance:adr-skill-platform`・ヒアリングRound 1より*
- **REQ-402**: システムは Claude Code skills および Codex CLI skills の両プラットフォームで実行可能でなければならない 🔵 *ヒアリングRound 1より*
- **REQ-403**: システムの主体はプロンプト・markdown で構成し、TypeScript コードは（必要な場合に限り）補助スクリプトに留めなければならない 🔵 *ヒアリングRound 1「実装形態」より*
- **REQ-404**: システムの出力は常に日本語で生成しなければならない（入力言語にかかわらず） 🔵 *ヒアリングRound 1より*
- **REQ-405**: システムは医療・法律・宗教の権威として断定的な助言を出してはならない 🔵 *PRD「注意」・PRD「重要な方針」より*
- **REQ-406**: システムは仏教教義として厳密に確認できない内容を、教義として断定して出力してはならない 🔵 *PRD「重要な方針」より*
- **REQ-407**: システムは、ユーザー入力に対する善悪判断を即座に行ってはならず、まず観察可能な現象として扱わなければならない 🔵 *PRD「重要な方針」・codd Wave 3 規約「observation-first principle」より*
- **REQ-408**: システムは Skill 自身がツールを呼び出さない構成（プロンプトのみ）で動作可能でなければならない（codd の AI コマンドが `--tools ""` で起動するため） 🟡 *codd.yaml `ai_command` から妥当な推測*
- **REQ-409**: システムは明示呼び出し（ユーザーが skill 名を指名 / 「四相分析して」等の依頼）でのみ起動するよう description を設計しなければならない 🔵 *ヒアリングRound 2「起動トリガー」より*

## 非機能要件

### パフォーマンス

- **NFR-001**: システムは入力長に上限を設けず、LLM の context window 上限まで受け付けなければならない 🔵 *ヒアリングRound 3「入力仕様」より*
- **NFR-002**: 通常モードの出力は、要約表 + 4 セクション × 3〜5 文 + まとめ 1〜2 文の規模に収めなければならない 🔵 *PRD「出力の長さ」より*

### セキュリティ

- **NFR-101**: システムはユーザー入力に含まれる個人情報・センシティブ情報を skill 内に永続化してはならない（skill はステートレスなプロンプトであるため） 🟡 *skill 一般的な特性から妥当な推測*

### ユーザビリティ

- **NFR-201**: システムの出力は、読み手が「たしかにこの現象はそういう特徴を持ち、そういう働きをして、私にはそう現れていて、そういう条件に支えられていたのか」と納得できる質を備えなければならない 🔵 *PRD「良い分析の基準」より*
- **NFR-202**: 出力は仏教未習の読み手にも理解可能なよう、パーリ語にはすべて現代語の言い換えを併記しなければならない 🔵 *PRD「納得感を出すための書き方」より*
- **NFR-203**: 心理的なテーマでは、責める口調ではなく、観察する口調で記述しなければならない 🔵 *PRD「納得感を出すための書き方」より*

### 配布・運用

- **NFR-301**: Skill は `skills/lakkhanadi-catukka/` ディレクトリ配下に集約し、Claude Code skills と Codex CLI skills の両方から参照可能な共通の SKILL.md を持たなければならない 🔵 *ヒアリングRound 4「配置構造」より*
- **NFR-302**: Skill のテストは LLM-as-judge による自動評価を備え、品質チェックリストに対する判定プロンプトを同梱しなければならない 🔵 *ヒアリングRound 3「テスト方針」・ヒアリングRound 4「judge MVP」より*

## Edgeケース

### エラー処理

- **EDGE-001**: 入力が空または空白のみの場合、システムは「分析対象を入力してください」という旨を返し、4 セクションの空虚な分析を生成してはならない 🟡 *既存PRD暗黙的方針から妥当な推測*
- **EDGE-002**: 入力が完全に non-Pali の意味不明文字列（例: ランダム文字列）の場合、システムは「対象として扱える事象が抽出できない」旨を返し、ユーザーに対象の言い換えを促す 🟡 *PRD「対象の言い換え」要件から妥当な推測*
- **EDGE-003**: 入力が極端に長く（例: 数万トークン）コンテキスト上限を超える場合、システムは要点要約を促すか、最初の N 文字で打ち切る旨を明示しなければならない 🟡 *NFR-001・LLM 一般制約から妥当な推測*
- **EDGE-004**: 入力に複数の対象が並列に含まれる場合、システムは最も中心的と思われる 1 対象を選んで分析し、他は対象外とする旨を明示する 🔵 *ヒアリングRound 2「複数対象=不要（1依頼1対象）」より*

### 境界値

- **EDGE-101**: 詳説セクションは最小 3 文、最大 8 文の範囲で生成されなければならない 🔵 *PRD「出力の長さ」より*
- **EDGE-102**: 仏教古典での扱いが両義的なトピック（仏教用語と現代心理用語の中間）については、応用的分析マーカーを付けたうえで「古典では〜とされ、現代的には〜と扱える」と両方の文脈を併記してよい 🟡 *REQ-105 から妥当な推測*

### 入力多様性

- **EDGE-201**: 入力対象が「文章」（例: 短編小説の抜粋）である場合、システムは文章全体ではなく文章が主題化している現象（怒り、孤独、希望など）を対象として抽出してから分析する 🟡 *PRD「文章表現などにも応用してよい」から妥当な推測*
- **EDGE-202**: 入力対象が「技術概念」（例: コンパイラ、TCP）である場合、システムは「四相分析の枠組みを応用すると」と明示し、技術現象を観察対象として扱う 🔵 *PRD「重要な方針」より*

## MVP スコープ

### Must Have（MVP）

- 四相分析コア出力（REQ-001〜010）
- 長さモード自動判定（REQ-101〜103）
- パーリ語正規化（REQ-107, REQ-108、既知リスト + IAST 揺れ）
- 応用的分析マーカー閾値判定（REQ-104, REQ-105）
- 警告と捉え提示（REQ-106）
- LLM-as-judge テスト（NFR-302）
- Codex CLI 対応パッケージ化（REQ-402）

### Out of Scope（MVP外）

- ファイル取り込み機能（テキスト引数のみで運用）
- 複数言語出力（日本語固定）
- 部分出力モード（lakkhaṇa だけ等）
- 出典・根拠を能動的に表示する機能（lakkhanadi-catukka-example.ncl 検索コマンド）
