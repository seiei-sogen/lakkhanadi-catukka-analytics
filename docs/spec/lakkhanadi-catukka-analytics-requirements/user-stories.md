## エピック 1: 四相分析コアの提供

仏教学習者・瞑想実践者・概念整理を求める一般ユーザーが、対象を四相の枠組みで観察できるようにする。

### ストーリー 1.1: 仏教古典トピックの正統な四相分析を得る 🔵

**信頼性**: 🔵 *PRD「目的」「重要な方針」・lakkhanadi-catukka-example.ncl 構造化リファレンスより*

**私は** 上座部アビダンマを学習しているユーザー **として**
**地・水・火・風や眼根などの古典トピックの四相を確認したい**
**そうすることで** 自分の理解が原典と整合しているかをチェックできる

**関連要件**: REQ-001〜010, REQ-104

**詳細シナリオ**:
1. ユーザーが skill を明示呼び出しし、「pathavī-dhātu の四相を分析して」と依頼する
2. システムは入力を「地大（pathavī-dhātu, earth-element）」と言い換える
3. システムは要約表（特相: 硬さ / 作用: 共生する物質の基盤となること / 現れ方: 共生物質を受け取るあり方 / 近因: 残り三大）を出力する
4. システムは各セクションを 3〜5 文で詳説し、応用的分析マーカーは付けない
5. システムは「一言でまとめると」で全体を 1〜2 文に集約する

**前提条件**:
- 入力対象が lakkhanadi-catukka-example.ncl または注釈伝統で四相が明示されているトピックであること

**制約事項**:
- 古典に沿った内容であり、教義として断定的な助言とならないこと

**優先度**: Must Have

---

### ストーリー 1.2: 心理現象を四相の枠組みで観察する 🔵

**信頼性**: 🔵 *PRD「重要な方針」・ヒアリングRound 2「応用的分析」より*

**私は** 自分の感情パターンを観察したいユーザー **として**
**「最近の漠然とした焦り」のような心理現象を四相分析してほしい**
**そうすることで** 焦りを責めずに観察対象として整理し、近接条件を理解できる

**関連要件**: REQ-001〜010, REQ-105, REQ-009

**詳細シナリオ**:
1. ユーザーが「最近の漠然とした焦りを四相分析して」と依頼する
2. システムは応用的分析マーカー（「四相分析の枠組みを応用すると」）を冒頭に付ける
3. システムは要約表で焦りの特相（時間との緊張関係）・作用（行動を促進し集中を妨げる）・現れ方（胸の浅い呼吸、急ぐ思考）・近因（締切・比較・身体疲労）を提示する
4. paccupaṭṭhāna セクションでは身体感覚 / 思考・感情・イメージ / 世界の見え方を具体的に書く
5. padaṭṭhāna セクションでは責める口調を避け、観察的に近接条件を列挙する

**前提条件**: なし

**制約事項**:
- 心理的なテーマで責める口調にならないこと（NFR-203）
- 医療・心理療法の権威として断定しないこと

**優先度**: Must Have

### ストーリー 1.3: 技術概念を四相の枠組みで言語化する 🟡

**信頼性**: 🟡 *PRD「技術概念にも応用してよい」から妥当な推測*

**私は** ソフトウェアエンジニアのユーザー **として**
**「TCP コネクション」「キャッシュミス」のような技術概念を四相分析したい**
**そうすることで** 技術現象の本質・働き・観察され方・近接条件を整理できる

**関連要件**: REQ-001〜010, REQ-105, EDGE-202

**詳細シナリオ**:
1. ユーザーが「TCP コネクションを四相分析して」と依頼する
2. システムは応用的分析マーカーを付与する
3. システムは技術現象を観察対象として扱い、特相 / 作用 / 現れ方 / 近因を技術文脈で記述する
4. パーリ語の技法的用語と現代技術用語の橋渡し説明を含めてもよい（REQ-301）

**優先度**: Should Have

**備考**: PRD の「技術概念にも応用してよい」を直接運用化したストーリー。アビダンマ用語と技術用語の混在の品質は、設計フェーズと judge プロンプトで担保する

## エピック 2: 出力品質と長さ制御

出力の品質を一貫して保ち、ユーザーの利用シーンに応じて長さを調整できるようにする。

### ストーリー 2.1: 短く要約して欲しい 🔵

**信頼性**: 🔵 *PRD「出力の長さ」より*

**私は** 急いでいるユーザー **として**
**「短く」「簡潔に」などのシグナルで要約表中心の出力を得たい**
**そうすることで** 即座に四相の概念マップを把握できる

**関連要件**: REQ-101

**詳細シナリオ**:
1. ユーザーが「忙しいので短く四相分析して」と依頼する
2. システムは長さシグナルを検知し、要約表中心の短い出力を生成する
3. 詳説は省略または 1〜2 文に圧縮する

**優先度**: Must Have

---

### ストーリー 2.2: 詳しい分析が欲しい 🔵

**信頼性**: 🔵 *PRD「出力の長さ」より*

**私は** 内省的に深掘りしたいユーザー **として**
**「詳しく」「納得感ある感じで」などのシグナルで各セクション 5〜8 文の出力を得たい**
**そうすることで** 自分の経験に即した具体例まで踏み込んだ理解ができる

**関連要件**: REQ-102

**詳細シナリオ**:
1. ユーザーが「詳しく、納得感ある感じで分析して」と依頼する
2. システムは長さシグナルを検知し、詳説モードへ切り替える
3. 各セクションを 5〜8 文に拡張し、具体例・対比・観点を増やす

**優先度**: Must Have

---

### ストーリー 2.3: 通常モードで安定した分析を得る 🔵

**信頼性**: 🔵 *PRD「出力の長さ」より*

**私は** 特に長さ指定をしていないユーザー **として**
**通常モード（要約表 + 各 3〜5 文の詳説 + まとめ）の標準出力を期待する**

**関連要件**: REQ-103, REQ-004

**優先度**: Must Have

## エピック 3: パーリ語正規化と古典整合

ユーザーがパーリ語を正確にタイプできなくても、標準表記で扱えるようにする。

### ストーリー 3.1: 既知の異形を自動修正する 🔵

**信頼性**: 🔵 *PRD「重要な方針」より*

**私は** パーリ語に自信がないユーザー **として**
**「padahāna」「paccupahāna」などと入力しても標準表記として扱ってほしい**
**そうすることで** 誤記を恐れずに学習を進められる

**関連要件**: REQ-107

**詳細シナリオ**:
1. ユーザーが「padahāna について四相分析して」と入力する
2. システムは padaṭṭhāna に正規化したうえで分析を実行する
3. 出力では標準表記を採用しつつ、必要に応じてユーザー入力の表記揺れに触れる

**優先度**: Must Have

---

### ストーリー 3.2: IAST／ダイアクリティカル揺れを吸収する 🔵

**信頼性**: 🔵 *ヒアリングRound 3「正規化スコープ」より*

**私は** 一般的なローマ字でパーリ語を入力するユーザー **として**
**「lakkhana」「paccupatthana」などダイアクリティカルなしの表記でも標準として扱ってほしい**

**関連要件**: REQ-108

**優先度**: Must Have

## エピック 4: 安全な対応と非権威的応答

医療・宗教・他者評価の要求があっても、観察的に応える。

### ストーリー 4.1: 重い悩みを観察対象として扱う 🔵

**信頼性**: 🔵 *PRD「注意」・ヒアリングRound 3「拒否ポリシー」より*

**私は** つらい状態のユーザー **として**
**「私は鬱なのか分析して」のような診断要求も、断られず観察的に扱ってほしい**
**そうすることで** 自分の状態を観察可能な現象として整理できる

**関連要件**: REQ-106, REQ-405, REQ-407

**詳細シナリオ**:
1. ユーザーが診断的な質問を投げる
2. システムは「医療・心理学的診断はできない」旨を冒頭で警告する
3. システムは対象を「現在感じている重さ」「日々のエネルギーの低下」など観察可能な現象に捉え直す
4. 捉え直した対象について四相分析を実行する
5. 終わりに「専門家への相談を促す」一文を含める（REQ-201 系）

**優先度**: Must Have

---

### ストーリー 4.2: 他者を裁く要求を避ける 🟡

**信頼性**: 🟡 *PRD「責める口調ではなく観察する口調」より妥当な推測*

**私は** 「あの人の性格を分析して」と尋ねるユーザー **として**
**他者の人格判定にはならず、自分が観察できる現象（怒り・避けたさ等）に向けてもらえる**

**関連要件**: REQ-106, NFR-203

**優先度**: Should Have
