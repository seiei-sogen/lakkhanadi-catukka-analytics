## タスク概要

Rust 補助スクリプト `normalize.rs`（前処理フック）が利用できないホスト環境でも LLM 自身が Pali 用語の異形・IAST/ダイアクリティカル揺れを正規化できるよう、SKILL.md に正規化対応表を記述する。`normalizer` モジュールの二段冗長設計（Rust スクリプトと同等の効果を Markdown 内テーブルで再現）の片翼を担う。極端な崩れは「不明な表記」として扱う方針も明示する。

## 実装詳細

### 1. 正規化テーブルセクションの位置 🔵

**信頼性**: 🔵 *dataflow.md フロー 5「正規化 → SKILL.md 本体処理へ」より*

「入力種別判定」（TASK-0006）の直後、「四相生成チェックリスト」（TASK-0011）の前に配置する。長さモード判定・安全性判定・古典 vs 応用判定（TASK-0007〜0009）は内容生成方針なので、正規化はそれらより前に走る前処理段とみなす。

### 2. 既知異形対応表 🔵

**信頼性**: 🔵 *PRD「重要な方針」・REQ-107・interfaces.ts PaliNormalizationRule より*

Markdown テーブル形式で `from / to / source` の 3 列を記述する。

```
| 異形 (from)    | 標準表記 (to)    | 出典      |
|----------------|------------------|-----------|
| padahāna       | padaṭṭhāna       | REQ-107   |
| paccupahāna    | paccupaṭṭhāna    | REQ-107   |
| ...            | ...              | ...       |
```

実際の異形リストは `lakkhanadi-catukka-example.ncl` の examples / pali_compact フィールドおよび PRD で言及される全異形を網羅する。

### 3. IAST/ダイアクリティカル対応表 🔵

**信頼性**: 🔵 *REQ-108・ヒアリングRound 3「IAST 揺れ吸収」より*

ダイアクリティカル記号（ṇ, ṭ, ṃ, ā, ī 等）が抜けた表記を標準表記に変換する規則を記述。

```
| 揺れ (from)      | 標準表記 (to)      | 出典      |
|------------------|--------------------|-----------|
| lakkhana         | lakkhaṇa           | REQ-108   |
| paccupatthana    | paccupaṭṭhāna      | REQ-108   |
| thana            | ṭhāna              | REQ-108   |
| anatta           | anattā             | REQ-108   |
| ...              | ...                | ...       |
```

### 4. 正規化適用ルール 🔵

**信頼性**: 🔵 *dataflow.md フロー 5・REQ-107, REQ-108 より*

- 入力テキスト中で対応表の異形を検出したら、内部処理および出力で標準表記を採用する
- 大文字小文字は区別しない（実装裁量）🟡
- 文中の複数同時出現にすべて適用する
- 出力中で初出時に「padaṭṭhāna（近因）」のように現代語併記する（REQ-006 と整合）

### 5. 極端な崩れの扱い 🔵

**信頼性**: 🔵 *ヒアリングRound 3「推測まで」は不採用・dataflow.md フロー 5「極端な崩れ → 不明な表記」より*

- "padaaaaana"、"padaTthana123" のような対応表に該当しない極端な崩れは正規化しない
- 「不明な表記」として扱い、ユーザーに「標準表記での再入力」を促すか、対象として扱える事象がない旨を返す（EDGE-002 と接続）
- 推測による補正は行わない（誤った正規化を避ける）

### 6. 二段冗長の意図コメント 🟡

**信頼性**: 🟡 *architecture.md「補助構成: Rust 前処理スクリプト」備考より*

SKILL.md 内のコメント（HTML コメント等）で、Rust スクリプト `normalize.rs` がホスト前処理段で利用可能な場合と、利用できない環境で LLM 自身が本テーブルを参照する場合の両方が想定されることを記載する。実装フェーズで Rust スクリプトと本テーブルのルール差分が生じないよう、`scripts/normalize.rs` 側の `NORMALIZATION_RULES` と必ず同期する旨を明記する。
