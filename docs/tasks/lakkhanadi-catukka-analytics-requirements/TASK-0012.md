## 実装詳細

### 1. セクション順序の最終確定 🔵

**信頼性**: 🔵 *dataflow.md「システム全体のデータフロー」より*

最終的な SKILL.md のセクション順は以下：

```
1. YAML フロントマター（name, description）       … TASK-0004
2. 役割宣言・全体方針・拒否ポリシー                … TASK-0004
3. 入力種別判定（empty/gibberish/over-context/...） … TASK-0006
4. Pali 表記正規化テーブル                        … TASK-0010
5. 長さモード判定                                  … TASK-0007
6. 安全性判定 + 警告/観察的捉え直し                … TASK-0008
7. 古典 vs 応用判定                                … TASK-0009
8. 四相生成チェックリスト + 観点網羅指示           … TASK-0011
9. 出力テンプレート（言い換え→要約表→詳説→まとめ）… TASK-0005
10. 一言まとめ生成ガイド                           … TASK-0005
```

### 2. 相互参照の整合性確認 🔵

**信頼性**: 🔵 *各タスクの依存関係から*

- TASK-0007（長さモード）→ TASK-0011（四相生成）の文数指示が一致しているか
- TASK-0008（安全性）→ TASK-0011（四相生成）で警告 + 観察的捉え直し後の四相生成手順が連結しているか
- TASK-0009（古典 vs 応用）→ TASK-0011 でマーカー付与判定が活用されているか
- TASK-0010（正規化）→ TASK-0005（出力テンプレート）で標準表記が採用されているか
- TASK-0006（入力種別判定）→ EDGE 系の応答テンプレートが TASK-0005 と矛盾しないか

### 3. 矛盾箇所の解消 🟡

**信頼性**: 🟡 *統合時に発見される可能性のある競合の妥当な推測*

- 文数指示のダブり（TASK-0005, TASK-0007, TASK-0011）→ 単一の正規箇所（TASK-0007 のモード判定セクション）に集約し他は参照のみ
- 観察的トーン指示のダブり（TASK-0008, TASK-0011）→ 全体方針（TASK-0004）に総則として 1 箇所、各論で参照
- 競合シグナル（短く + 詳しく）の扱いが TASK-0007 と TASK-0011 で一貫しているか

### 4. 冗長削減 🟡

**信頼性**: 🟡 *レビュー段階で発生する可能性のある実装裁量*

3 箇所以上で繰り返される指示は 1 箇所に集約して相互参照する。SKILL.md のトークン数増大を避ける。Markdown のヘッダ階層を整理し、セクション間の依存関係を視覚化する。

### 5. judge プロンプトでのフル評価 🔵

**信頼性**: 🔵 *NFR-302・dataflow.md フロー 7 より*

7 サンプル（TASK-0016 で作成）× 8 観点 = 56 判定で、各観点が pass / fail / n/a を期待通りに返すかを確認する。fail がある場合は SKILL.md の該当セクション（TASK-0004〜0011 のいずれか）にフィードバックして修正。

### 6. プラットフォーム互換性の最終確認 🔵

**信頼性**: 🔵 *REQ-402・NFR-301・TASK-0003 で確定したフロントマター仕様より*

- YAML フロントマターが両プラットフォーム最小共通項（name, description）に絞られているか
- Markdown 構造が Claude Code skills / Codex CLI skills 両者で読み込み可能か（特殊な拡張記法を使っていないか）
- ファイル末尾の改行・BOM 等の互換性問題がないか

## 単体テスト要件

### テストケース1: 全 7 サンプル × 8 観点の judge 評価 🔵

**信頼性**: 🔵 *NFR-302・TASK-0017 で構築する judge より*

**Given**: 統合済み SKILL.md と TASK-0016 の 7 サンプル
**When**: 各サンプルを SKILL.md に通し、出力を judge プロンプトで評価
**Then**: すべての観点で `verdict: 'pass'` または `verdict: 'n/a'`、`overallPass: true`

### テストケース2: 競合シグナル時の一貫した挙動 🟡

**信頼性**: 🟡 *TC-LEN-E01・PRD 未定義の妥当な解釈*

**Given**: 入力「短く詳しく 怒り を分析して」
**When**: SKILL.md の長さモード判定 + 四相生成
**Then**: 一貫した方針（後続シグナル優先 or 通常モードフォールバック）で出力される

### テストケース3: 競合フェーズ間の連結 🔵

**信頼性**: 🔵 *dataflow.md「全体フロー」より*

**Given**: 入力「私は鬱なのか四相分析して」（医療診断要求）
**When**: 安全性判定 → 古典/応用判定 → 四相生成 の順に処理
**Then**: 警告 + 観察的捉え直し（TASK-0008）→ 応用マーカー（TASK-0009）→ 観察的トーンで四相生成（TASK-0011）→ 専門家相談促進が末尾（TASK-0008）の順で出力される
