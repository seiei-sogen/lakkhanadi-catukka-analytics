## 実装詳細

### 1. ルールテーブル基本構造 🔵

**信頼性**: 🔵 *interfaces.ts `PaliNormalizationRule` インターフェース定義より*

`&[PaliNormalizationRule]`（const 配列スライス）として定義し、配列順を優先順位とする（前のものから先に適用）。

```rust
// skills/lakkhanadi-catukka/scripts/normalize.rs

#[derive(Debug, Clone, Copy, PartialEq, Eq)]
pub enum Category {
    KnownVariant,
    IastDiacritic,
}

#[derive(Debug, Clone, Copy)]
pub struct PaliNormalizationRule {
    pub from: &'static str,
    pub to: &'static str,
    pub category: Category,
    pub source: &'static str,
}

pub const NORMALIZATION_RULES: &[PaliNormalizationRule] = &[
    // known-variant（PRD 由来既知異形）
    // iast-diacritic（IAST/ダイアクリティカル揺れ）
];
```

### 2. 既知異形（known-variant） 🔵

**信頼性**: 🔵 *PRD「重要な方針」・REQ-107・dataflow.md フロー 5 R1 より*

最低限以下を含める:

| from | to | source |
|------|----|----|
| padahāna | padaṭṭhāna | REQ-107 / TC-NORM-01 |
| paccupahāna | paccupaṭṭhāna | REQ-107 / TC-NORM-02 |

その他、PRD・lakkhanadi-catukka-example.ncl の examples / pali_compact で言及される異形があれば追加する。

### 3. IAST/ダイアクリティカル揺れ（iast-diacritic） 🔵

**信頼性**: 🔵 *REQ-108・TC-NORM-03・dataflow.md フロー 5 R1 より*

最低限以下を含める:

| from | to | source |
|------|----|----|
| lakkhana | lakkhaṇa | REQ-108 / TC-NORM-03 |
| paccupatthana | paccupaṭṭhāna | REQ-108 / TC-NORM-03 |
| thana | ṭhāna | REQ-108 |
| anatta | anattā | REQ-108 |

その他、四相分析で頻出するパーリ語のダイアクリティカルなし表記を追加する（dhatu→dhātu, citta→citta は同形なので不要 等、判別して登録）。

### 4. 大文字小文字の扱い指針 🟡

**信頼性**: 🟡 *interfaces.ts コメント「大文字小文字を区別しないマッチを行うことを推奨」・実装詳細は妥当な推測*

- ルールテーブル自体は小文字で `from` を定義する
- 関数実装側（TASK-0014）で case-insensitive マッチを行う方針をルール側コメントで明記
- 大文字始まり（`Lakkhana`）も同じルールでマッチする想定

### 5. ルール優先順位の整理 🔵

**信頼性**: 🔵 *interfaces.ts コメント「配列順 (前のものから先に適用)」より*

- 既知異形 → IAST 揺れの順で登録（既知異形の方が優先度高）
- 部分一致衝突を避けるため、長い from を先に置く（例: `paccupatthana` は `thana` より先）

### 6. メタ情報 🔵

**信頼性**: 🔵 *interfaces.ts `source` フィールド「REQ-XXX, TC-NORM-XX 参照」より*

- 各ルールに `source: "REQ-107"` 等のメタ情報を必ず付与
- judge / デバッグ時の追跡を可能にする

## 単体テスト要件

### テストケース1: ルールテーブル整合性 🔵

**信頼性**: 🔵 *interfaces.ts `PaliNormalizationRule` 型と REQ-107/108 より*

**Given**: `NORMALIZATION_RULES` 配列
**When**: 配列を走査する
**Then**:
- 配列が空でない
- すべてのエントリで `from != to`
- `from` に重複がない
- 各エントリの `category` が `Category::KnownVariant` または `Category::IastDiacritic` である
- 各エントリの `source` が空文字でない

### テストケース2: 必須ルールの存在確認 🔵

**信頼性**: 🔵 *TC-NORM-01, TC-NORM-02, TC-NORM-03 より*

**Given**: `NORMALIZATION_RULES` 配列
**When**: 検索する
**Then**:
- `padahāna` → `padaṭṭhāna` のルールが存在する（known-variant）
- `paccupahāna` → `paccupaṭṭhāna` のルールが存在する（known-variant）
- `lakkhana` → `lakkhaṇa` のルールが存在する（iast-diacritic）
- `paccupatthana` → `paccupaṭṭhāna` のルールが存在する（iast-diacritic）

### テストケース3: 部分一致衝突の確認 🟡

**信頼性**: 🟡 *優先順位仕様の妥当な推測*

**Given**: 複数のルールが部分一致しうる場合
**When**: より長いマッチを優先するよう配列順が並んでいる
**Then**: `paccupatthana` のインデックス < `thana` のインデックス
