## 実装詳細

### 1. 関数シグネチャ 🔵

**信頼性**: 🔵 *interfaces.ts `export type Normalize = (input: string) => string` より*

```ts
import type { Normalize, PaliNormalizationRule } from '../../../docs/design/lakkhanadi-catukka-analytics-requirements/interfaces';
import { NORMALIZATION_RULES } from './normalize-rules'; // または同一ファイル内定義

export const normalize: Normalize = (input) => {
  // ルールを配列順で順次適用
};
```

または

```ts
export function normalize(input: string): string { ... }
```

ESM 互換のエクスポート形式とする。

### 2. ルール適用ロジック 🔵

**信頼性**: 🔵 *dataflow.md フロー 5 R1 / interfaces.ts コメントより*

- `NORMALIZATION_RULES` を配列順に走査
- 各ルールについて、`from` を正規表現に変換して入力中の出現を `to` に置換
- グローバル + case-insensitive フラグで全置換

```ts
let result = input;
for (const rule of NORMALIZATION_RULES) {
  const pattern = new RegExp(escapeRegExp(rule.from), 'gi');
  result = result.replace(pattern, rule.to);
}
return result;
```

### 3. case-insensitive マッチ 🟡

**信頼性**: 🟡 *interfaces.ts コメント「大文字小文字を区別しないマッチを行うことを推奨」・実装詳細は妥当な推測*

- 正規表現フラグ `i` を使う
- 入力 `Lakkhana` も `lakkhaṇa`（標準表記、小文字）に置換される
- 大文字保持が必要な場合の方針は実装コメントで明記する（MVP では小文字標準表記に統一）

### 4. 正規表現の境界条件 🟡

**信頼性**: 🟡 *部分一致衝突を避けるための妥当な推測*

- 単純な部分一致でよい（単語境界 `\b` は IAST 文字を考慮しないため使わない方針）
- ルール配列順で長いものを先に置くことで部分一致衝突を回避（TASK-0013 で担保）
- 例: `paccupatthana` を先に置換すると、後続の `thana` ルールが残骸を変換することはない

### 5. 極端な崩れのスルー 🔵

**信頼性**: 🔵 *interfaces.ts コメント「該当しない極端な崩れ ("padaaaaana" 等) はそのまま通し、下流の SKILL.md (LLM) 側で "unknown_term" として扱わせる」より*

- ルールテーブルに該当しない入力は変換せずそのまま返す
- `normalize('padaaaaana')` → `'padaaaaana'`
- 「推測まで」の自動修正は実装しない（dataflow.md フロー 5 備考）

### 6. 正規表現エスケープ補助 🔵

**信頼性**: 🔵 *汎用的な実装パターン*

`from` 文字列に正規表現メタ文字が含まれる場合に備えて `escapeRegExp` ヘルパを用意する（IAST 文字には影響しないが防御的に）。

```ts
function escapeRegExp(s: string): string {
  return s.replace(/[.*+?^${}()|[\]\\]/g, '\\$&');
}
```

### 7. ESM エクスポート形式 🔵

**信頼性**: 🔵 *TASK-0002 `tsconfig.json module: ESNext/NodeNext` より*

- `export const normalize` または `export function normalize` で名前付きエクスポート
- default export は使わない（vitest テストや SKILL.md からの参照を明示的にする）
